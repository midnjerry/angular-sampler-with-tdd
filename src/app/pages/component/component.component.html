Angular Component

Components should be treated as mini-programs or widgets that can be generally used for a specific functionality. Imagine these components as custom "HTML tags" that you create for yourself or for your team.

For example, let's make a CodeBlock component that will allow us to format code into a more readable format using colorized text to highlight keywords. Let's make a <codeblock> tag that we can use to surround our target code.

For example, if we write this as html

<codeblock language="language-html">
{{htmlCode1}}
</codeblock>

we should display this:

{{htmlCode1}}

This would be great! Now, as we create more and more content, we can always reuse this <codeblock> tag to highlight code! Angular allows us to granularize our functionality into small components that we can share and reuse.

Quick Notes

Every component has the following files when generated:

  • .ts file
    • The "brain" of the component
    • This is the Typescript class that holds programmatic detail of the component.
    • Constructor is called from here
    • Angular lifecycle event handlers (methods) can be implemented here
  • .spec.ts file
    • Test file for unit and integration testing
  • .html file
    • The HTML for the component view is housed here.
  • .css file
    • Placeholder stylesheet to customize the component

@Component decorator (aka annotation)

  • selector: The name of your "custom tag".
    • Ex. A component with "selector: app-navbar" can be referenced in html with: <app-navbar>
  • templateUrl: Html file used for component display.
  • styleUrls: Stylesheets to use for component. Angular encapsulates the stylesheet so they are not treated globally.
  • Use the @Input() decorator to accept parameters within the first tag.
    • In the example above, that's the "language" attribute.
